# Ooru Logix Operational Playbooks

This folder contains the complete operational backbone for Ooru Logix ShopSense — the documents that make the business run consistently.

These implement the recommendations from:
1. **McKinsey Report**: "Execute brutal beachhead strategy, 15 units in 6 months, de-risk vision model on real data"
2. **Claude Analysis**: "Get 1 paying customer before anything else. Build hardware support playbook NOW. Solo founder will hit a wall at 10-15 deployments."

---

## Files & Purpose

### 1. Business_Operating_Rhythm.md
**The daily/weekly/monthly operating system for solo founder.**

Use this to structure your week so you stay focused on what matters:
- Daily ritual (30 min): Check pipeline, do focus work, log progress
- Weekly review (1 hour): Check revenue vs target, close decision gates
- Monthly review (2 hours): Full P&L, unit economics, roadmap check
- Decision filter: Every task/request goes through a 5-question filter

**When to use:** Every Monday (weekly review), first Sunday of month (monthly review), daily (ritual + decision filter)

---

### 2. Customer_Success_Playbook.md
**What happens AFTER a ShopSense system is deployed.**

This prevents churn, generates referrals, and builds testimonials for fundraising. Covers:
- Day 0 Installation: Pre-checklist, procedure, go-live verification
- Day 1 Check-in: First customer call, remote monitoring
- Week 1-2: Data gathering, pilot assessment, conversion conversation
- Post-conversion: Monthly support cadence, model updates, referral asks
- Health score: Simple system to flag at-risk customers early

**When to use:** For every new pilot (from installation through first 3 months of support)

---

### 3. Hardware_Support_Runbook.md
**The "3 AM Pi is dead and the shop opens at 5 AM" playbook.**

This prevents the solo founder bottleneck. Covers:
- Spare parts inventory (what to keep on hand)
- P0/P1/P2 failure scenarios with step-by-step resolution
- Remote support protocol (SSH, health checks, logs)
- Escalation rules (when to dispatch on-site vs handle remote)
- Warranty policy + preventive maintenance schedule

**When to use:** When troubleshooting any hardware issue. Reference by P0/P1/P2 level and symptom.

---

### 4. Sales_Pipeline_SOP.md
**Standard Operating Procedure for converting shops to paying customers.**

Complete guide from cold shop to paying customer. Covers:
- Stage 1 (Identification): Target shop profile (ICP), how to find them
- Stage 2 (First Contact): Opening script, demo offer, tracking
- Stage 3 (Demo): Demo setup, agenda, soft close
- Stage 4 (Pilot): 14-day trial mechanics
- Stage 5 (Conversion): ROI conversation, close, invoice
- Pipeline health rules + success metrics

**When to use:** When identifying shops, preparing demos, or following up with prospects. Update pipeline weekly.

---

### 5. Investor_Readiness_Scorecard.md
**Exactly WHEN and HOW to approach investors.**

Core principle: Do NOT pitch until you have 3+ paying customers. This doc defines:
- Blocking requirements (6 categories, must be 100% complete)
- Nice-to-haves (helpful but not required)
- Target investors (tier 1 angels → tier 2 hardware VCs)
- 30-second pitch templates (Version 1/2/3 as you grow)
- Term sheet expectations
- Questions investors will ask
- Monthly readiness review checklist

**When to use:** Month 4+ when you have 3+ customers. Use the scorecard to track progress toward fundraising readiness.

## How to Implement This System

### Month 1 (Now - April 2026)
- [ ] Print or bookmark all 5 playbooks
- [ ] Start Business_Operating_Rhythm (daily ritual + weekly review)
- [ ] Use Sales_Pipeline_SOP to structure your shop outreach
- [ ] Reference Customer_Success_Playbook for first pilot (if you have one)

### Month 2-3 (May-June)
- [ ] Deploy 3-4 pilots using playbooks #2 and #4
- [ ] Track unit economics against Hardware_Support_Runbook
- [ ] Continue weekly reviews (Business_Operating_Rhythm)
- [ ] Start Investor_Readiness_Scorecard tracking (even though not fundraising yet)

### Month 4+ (July onward)
- [ ] Convert first 3 pilots to paying customers
- [ ] Use Investor_Readiness_Scorecard to guide final preparations
- [ ] When scorecard is 100% green, start angel network intros

## Anti-Patterns These Playbooks Prevent

1. **Writing docs instead of talking to customers** → Sales_Pipeline_SOP forces you to identify + contact 2-3 shops weekly
2. **Building features for imaginary customers** → Business_Operating_Rhythm decision filter stops this
3. **Pursuing web design clients instead of ShopSense** → Operating rhythm gates non-ShopSense work until Gate 3 is closed
4. **Planning fundraising before 3 customers** → Investor_Readiness_Scorecard blocks fundraising until you earn it
5. **Solo founder hitting a wall at 10-15 units** → Hardware_Support_Runbook enables remote-first support at scale

## Questions?

These playbooks are living documents. Update them as you learn:
- Customer_Success_Playbook: Add new issues as they arise, document solutions
- Hardware_Support_Runbook: Record actual failure times + fixes
- Sales_Pipeline_SOP: Document objections you hear, refine responses
- Business_Operating_Rhythm: Adjust review cadence if weekly isn't working
- Investor_Readiness_Scorecard: Track your actual progress vs targets

**Golden rule:** If you notice something's broken in a playbook, fix it. These are YOUR operating procedures, not gospel.
